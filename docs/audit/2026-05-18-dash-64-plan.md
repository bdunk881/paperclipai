# DASH-64 — controlPlaneStore Map-drop plan

**Owner:** Claude / Codex collaboration (DASH initiative)
**Parent:** [`docs/audit/2026-05-18-postgres-migration-plan.md`](2026-05-18-postgres-migration-plan.md)
**Status:** Plan — break into the seven sub-PRs below.

## Why this is its own plan

DASH-64 was deferred out of the parent postgres-migration plan because
`src/controlPlane/controlPlaneStore.ts` is one 3,100-line file holding
**14 top-level Maps** and a long tail of synchronous read methods used by
seven distinct surfaces (routes, HITL, observability, reports, engine,
tests). A single PR that drops all 14 maps would cascade into roughly
50 call sites across the repo and cannot be rolled back independently.

The right shape is one PR per entity. Each PR drops one Map (or a small
cluster of related Maps), converts the matching read methods to async,
and updates every caller. The execution-state entities (tasks,
heartbeats, spend, budget alerts) already have a finished
`controlPlaneRepository` so those PRs are mostly mechanical. The
canonical-domain entities (teams, agents, executions, companies) need
new read helpers but the write paths already exist.

## Strategy

**Drop the Maps, make reads async, route every read through Postgres** —
do not try to extend the DASH-27 / DASH-40 hybrid pattern. The hybrid
pattern was a stopgap when the in-memory rows carried fields the
canonical Postgres rows didn't (runtime-only fields like
`lastHeartbeatStatus`). After this sweep the canonical rows ARE the
source of truth; runtime-only fields either live as separate columns
on the canonical row or move to a derived view.

This is more invasive than route-layer glue, but it removes the
two-source-of-truth problem permanently and lets us delete the hybrid
fallback comments. The hybrid pattern was always meant to bridge the
gap to DASH-64; this is DASH-64.

### Sequencing principle

Land the sub-PRs in the order that minimises rework:

1. **First**: execution-state entities (tasks, heartbeats, spend, budget
   alerts) — repository already done, only the store + callers need to
   change. Each is small.
2. **Then**: executions — the missing repo read helper is the only new
   code; everything else mirrors the execution-state pattern.
3. **Then**: agents — touches `agentRoutes.ts` (the agentCap entitlement
   gate is the highest-traffic path; do it after the easier wins so the
   pattern is proven).
4. **Then**: teams — touches the most surfaces (`getTeam` is in
   `controlPlaneRoutes`, `hitlStore`, `reportRoutes`).
5. **Last**: company suite — `companies` + `companyWorkspaces` +
   `companySecretBindings` + `companyIdempotencyIndex` collapse into
   one PR; they're tightly coupled (every write touches all four).
6. **Cleanup**: drop the three derivable index Maps (`teamWorkspaceIds`,
   `teamCompanyIds`, `companyTenantWorkspaceIds`) and the
   `ensureWorkspaceHydrated` boot-time hydration helper.

## Sub-PR breakdown

Each sub-PR ships independently against `dev`. Each runs the full CI
matrix and has its own Linear ticket (filed when this plan is approved).

### DASH-64.1 — Drop the `tasks` Map (S)

**Change**:
- Replace `listTasks` and all internal task reads in
  `controlPlaneStore.ts` with `await controlPlaneRepository.listTasks(...)`.
- Convert `controlPlaneStore.listTasks` signature from sync to async.
- Update callers: `controlPlaneRoutes.ts:571`, `:796`, `hitlStore.ts:509`,
  `observability/service.ts:222`, `reportRoutes.ts:117`,
  `stepHandlers.test.ts:871`, `controlPlaneStore.test.ts:76`.
- Delete the `tasks` Map declaration (line 294).
- Drop the in-memory `tasks.set(...)` writes — repository already
  writes through `controlPlaneRepository.upsertTask`.

**Risk**: Low. The repository has been in production since ALT-2042
(migration 019). Tests already cover both paths via the
`controlPlaneStore.hydration.rls.test.ts` suite.

**Effort**: ~150 lines changed across 7 files.

### DASH-64.2 — Drop the `heartbeats` Map (S)

### DASH-64.3 — Drop `spendEntries` + `budgetAlerts` Maps + spend snapshot (M)

**Change**:

**Effort**: ~180 lines across 4 files.

**Effort**: ~250 lines across 8 files.

### DASH-64.5 — Centralise canonical agent reads (drop the `agents` Map) (M)

### DASH-64.6 — Centralise canonical team reads (drop the `teams` Map) (M-L)

**Change**:
- Lift `loadCanonicalTeamDetail` (currently in `controlPlaneRoutes.ts:411`)
  into `controlPlaneRepository.getTeam(ctx, teamId)` +
  `controlPlaneRepository.listTeams(ctx)`.
- Replace `listTeams`, `getTeam`, `getMissionState` in the store with
  async repo calls. `getMissionState` builds from the team row, so its
  signature changes too.
- Callers: `controlPlaneRoutes.ts:310`, `:568` (the DASH-40 fallback
  block disappears — the canonical path becomes the only path),
  `hitlStore.ts:506`, `reportRoutes.ts:109`.
- `teamWorkspaceIds` cache is NOT deleted by this sub-PR.
  `workspaceContextForTeam` resolves a team's workspace from a cold
  path (no workspace context set yet), and `agent_teams` has FORCE
  RLS — a normal SELECT returns zero rows without
  `app.current_workspace_id`. The cache MUST stay reachable via
  migration 030's `lookup_team_workspace_id` SECURITY DEFINER helper.
  DASH-64.8 only drops the in-memory Map; the helper-backed fallback
  stays.
- Delete the `teams` Map.

**Risk**: Medium-high. `getTeam` is called from HITL, reports, and the
team detail page — three different request paths. Test the surface
end-to-end before merge.

**Effort**: ~400 lines across 6 files (includes deleting the
`loadCanonicalTeamDetail` block from `controlPlaneRoutes.ts`).

### DASH-64.7 — Collapse the company suite onto Postgres reads (L)

**Change**:
- New repo: `provisionedCompanyRepository` (or extend
  `controlPlaneRepository`) with:
  - `getProvisionedCompany(ctx, companyId)`
  - `listProvisionedCompanies(ctx)`
  - `getProvisionedWorkspace(ctx, workspaceId)`
  - `getCompanySecretBindings(ctx, companyId)`
  - `getCompanyByIdempotencyKey(ctx, workspaceId, userId, key)` —
    replaces `companyIdempotencyIndex` Map. The persisted constraint
    on `companies` is `(workspace_id, user_id, idempotency_key)` (see
    migration 013); the replacement query MUST scope by workspace AND
    user, NOT just `WHERE idempotency_key = $1` — otherwise a repeated
    key from another user/workspace looks like a collision and can
    either replay another tenant's row or fail provisioning.
  - The secrets reads use `provisioned_company_secrets`
    (migration 017's table; `secretsRepository.setSecrets` already
    writes to it).
- Replace every `companies.get` / `companyWorkspaces.get` /
  `companySecretBindings.get` / `companyIdempotencyIndex.get` inside
  `controlPlaneStore.ts` with await calls.
- Drop the four Maps (#6, #7, #8, #9).
- `companyTenantWorkspaceIds` (Map #14) goes away — derivable from
  the provisioned-company row's `workspace_id` column (NOT from
  `control_plane_company_lifecycle`, which is keyed by user_id and
  has no workspace_id).

**Risk**: High. This is the provisioning critical path. The
idempotency key check (DASH-43–style replay protection) MUST be exact
— a regression here means duplicate-provisioning bugs.

**Effort**: ~600 lines across 4 files. Single largest PR in the
sweep.

### DASH-64.8 — Delete the derivable indexes + the hydration helper (S)

**Change**:
- Delete `teamCompanyIds` and `companyTenantWorkspaceIds` Maps + every
  line that writes them. (Both are derivable from the persisted rows
  — see the inventory table above for the source columns.)
- Keep `teamWorkspaceIds`'s lookup path. The Map itself goes away,
  but `workspaceContextForTeam` must continue to resolve a team's
  workspace from a cold path. The `lookup_team_workspace_id`
  SECURITY DEFINER helper (migration 030) stays the production
  source of truth; the function should call it instead of reading
  the in-memory cache. Without this, cold-path task / heartbeat /
  spend writes that start from a teamId will skip their downstream
  persists (Codex review of this plan flagged the regression).
- Delete `ensureWorkspaceHydrated` (line 906) and every call site —
  the function no longer has anything to hydrate.
- Delete `hydratedWorkspaceUsers` Set (line 317).
- Verify CI Map allowlist check (DASH-65 grep guard) goes clean — no
  more `// allowlist:` comments needed for hybrid-mirror Maps in this
  file.
- Verify `controlPlaneStore.hydration.rls.test.ts` still passes — at
  this point it's testing the repository, not the hydration helper,
  so the test name might want to change too.

**Risk**: Low. By the time we get here every Map is already gone; this
is just cleanup of the scaffolding. The one non-trivial bit is keeping
`lookup_team_workspace_id` reachable from `workspaceContextForTeam` —
the test suite already covers that path.

**Effort**: ~150 lines across 3 files.

  - `controlPlaneStore.identityScope.test.ts` — same.

    "rows survive a process restart" against the repository.

## Rollout

- Each sub-PR runs the full CI matrix (lint, test-backend, test-api,
  test-api-integration, test-frontend, build-frontend, auth-regression,
  e2e, docker build-check).
- Auto-merge each on green; no overnight pause.
- After DASH-64.8 merges, **delete the
  `loadCanonicalTeamDetail` block from `controlPlaneRoutes.ts`** (it
  was only there as the route-layer-glue stopgap) and **delete the
  DASH-27 merge block from `agentRoutes.ts`** — both are dead code
  once the store reads canonical itself.
- Final verification: boot a fresh Fly machine, hit
  `GET /api/control-plane/teams`, `GET /api/agents`, run a workflow,
  verify the team page renders. Restart the machine, repeat — no
  empty-state regressions.
