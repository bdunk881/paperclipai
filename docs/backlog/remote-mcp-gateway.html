Backlog · Product brief · DASH-25

Remote MCP Gateway

Stop wiring OAuth one integration at a time. Become the workspace's single MCP-spec endpoint, expose every integration through it once, let every LLM provider (and the user's own Claude / ChatGPT) inherit the connection for free.

Status: backlogged behind the four agentic-infra PRs (save_memory tool, integration allowlist, OpenAI streaming/tool-loop parity, unified Credentials tab) which all shipped. This brief is the visualization. Implementation lands once those four are stable in dev.

The problem

Today, every integration AutoFlow supports is a separate OAuth wire-up: we register an app with Slack, an app with HubSpot, an app with Stripe, store tokens per workspace, refresh them, run the callbacks, etc. The Integrations rework (DASH-12/13) made the surface usable, but the underlying cost is still per-provider.

That cost scales linearly with our integration count, and worse: when a customer connects Slack to Claude Desktop OR uses a custom GPT with Slack actions, our AutoFlow connection is duplicate work. They had to OAuth twice. We can't see their Claude/ChatGPT side. They can't use AutoFlow-managed integrations from inside their Claude Desktop session.

The shift

Anthropic shipped mcp_servers support in the Messages API (Nov '25). OpenAI shipped the equivalent through the Responses API's tools: [{type: "mcp", server_url}]. Both providers can now call remote MCP servers — the provider hosts the round-trip; we just supply a URL plus a workspace-scoped auth token.

That means we don't need the LLM to call our integration through AutoFlow's own custom tool plumbing. We expose every integration as a remote MCP endpoint, and the provider's own loop calls it.

What changes in the AutoFlow product

Integrations page

Each row gets a second green pill: "Exposed as MCP". That state means we've issued a workspace-scoped JWT and the endpoint is live at mcp.helloautoflow.com/<integration>/<workspaceId>.

A new action becomes available alongside the existing Connect: "Use in Claude Desktop" and "Use in ChatGPT". Click → copy the right snippet for their tool's config file.

Hiring plan

When the LLM drafts an agent that needs Slack, we no longer have to pre-register Slack as an AutoFlow connector tool. The agent's provider call passes mcp_servers: [{url: ".../slack/<workspaceId>", auth}] and the provider's tool loop calls it directly.

Per-agent allowed_integration_slugs (PR 2/4) still gates which gateway routes get included in mcp_servers.

Concrete sequence: "use Slack via Claude Desktop"

  1. User connects Slack to AutoFlow as today (one-time OAuth).
  2. Integrations page now shows a second pill: Exposed as MCP.
  3. User clicks "Use in Claude Desktop" button.
  4. Modal shows a one-liner to paste into claude_desktop_config.json: 
    {
  "mcpServers": {
    "autoflow-slack": {
      "url": "https://mcp.helloautoflow.com/slack/ws_8a0f...",
      "auth": { "token": "wsk_live_..." }
    }
  }
}
  5. User restarts Claude Desktop. Their local Claude can now call Slack via AutoFlow — same OAuth token, same audit trail, same workspace scope.
  6. Same URL pattern works for OpenAI's Responses API and the AutoFlow engine itself.

What we need to build

  1. Workspace-scoped JWT issuance. Per-workspace service-account token, rotatable from the Credentials tab, signed with a key only the gateway worker holds.
  2. Cloudflare Worker gateway. One Worker per integration (or one Worker with route-based dispatch). Each adapter:
    • Validates the inbound JWT, extracts workspace + integration.
    • Looks up the workspace's OAuth token from the vault.
    • Translates incoming MCP protocol messages into the integration's native HTTP API.
    • Returns MCP-spec JSON to the calling provider.
  3. OAuth vault lookup endpoint. Tight read-only Postgres function callable from the Worker. Per-workspace, per- integration. Caches in-Worker KV for low-latency reads.
  4. Dashboard surfaces. Integration rows show MCP status + the "Use in Claude" / "Use in ChatGPT" CTAs.
  5. Per-agent allowlist enforcement. When AutoFlow's own engine calls a provider with mcp_servers, only include URLs the agent's allowed_integration_slugs permits. (PR 2 plumbing reused.)
  6. Audit log on every gateway call. Workspace + agent (when known) + integration + tool name + bytes. Same audit_log table we already use.
Phase 1: workspace JWT + Slack adapter as proof Phase 2: HubSpot · Linear · Stripe · Gmail Phase 3: "Use in Claude/ChatGPT" CTAs Phase 4: customer-self-serve gateway URL rotation

What it doesn't unlock

Claude Desktop's existing MCP servers stay invisible to us. Those are configured in the user's local config file; they run on the user's machine; the API has no surface that lists them. We can't show "green because you already have this in Claude Desktop." We can only show "green because you connected it through us, and you can also point Claude Desktop at us."

ChatGPT custom GPT actions stay invisible. Same reason — they live in OpenAI's UI, not the API.

The product framing is therefore not "detect your existing connections." It's "connect once with AutoFlow, get every LLM + your local desktop tools for free."

Risks worth naming up front

  • Cloudflare Workers MCP performance. MCP is a JSON-RPC-over-HTTP protocol. Cold starts on Workers should be sub-50ms. If we hit latency on first call, KV-cache the workspace → OAuth-token lookup.
  • Provider MCP spec drift. Anthropic and OpenAI both implement the spec, but in slightly different shapes today. Build the adapter to the strict spec, not to any one provider's bend.
  • JWT rotation UX. A leaked workspace token is the blast radius for that workspace's integrations. Make rotation one-click from the Credentials tab + email the owner on rotation + show last-used-at per token.
  • Audit log volume. Every Claude tool call lands in audit_log. Index on (workspace_id, integration, created_at) and retire rows past 90d to a cold table.
