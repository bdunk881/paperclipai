# Persistence audit — store-by-store inventory (HEL-80)

Every AutoFlow store either reads from Postgres or falls back to a process-local
in-memory implementation. This document is the canonical inventory of those
stores plus the rules every new store must follow.

## Rules every store must follow

1. **Production never falls back.** A store with no Postgres connection in
   production is a fatal startup error, not a silent degradation. The
   `inMemoryAllowed()` helper in `src/db/postgres.ts` enforces this — it
   returns `false` whenever `NODE_ENV === "production"`, regardless of any
   other env var.

2. **Double-locked safety.** Even in dev/test, the in-memory fallback only
   trips when BOTH:
   - `NODE_ENV` is `development` or `test`, **AND**
   - `AUTOFLOW_ALLOW_INMEMORY` is exactly the string `"true"`.

   This is belt-and-suspenders against an operator accidentally inheriting
   `NODE_ENV=development` on a production host — two mistakes have to land
   for fallback to trip.

3. **Standard call pattern.** Every store call uses one of these shapes:

   ```ts
   import { inMemoryAllowed, isPostgresConfigured } from "../db/postgres";

   if (isPostgresConfigured()) {
     // Postgres path
     return await queryPostgres(...);
   }

   if (inMemoryAllowed()) {
     // In-memory fallback (dev/test only, double-locked)
     return inMemoryStore.read(...);
   }

   throw new Error("DATABASE_URL is required for ...Store outside development/test");
   ```

4. **Bootstrap is fail-fast.** `initializePersistence()` in `src/bootstrap.ts`:
   - Throws `PERSISTENCE_REQUIRED_ERROR` when neither Postgres nor in-memory is
     allowed (covers prod-without-DB).
   - Throws `POSTGRES_UNREACHABLE_PRODUCTION_ERROR` when Postgres IS configured
     but unreachable AND `NODE_ENV=production`. Dev/test continues with a
     warning so iteration isn't blocked by a flaky local DB.

## Adding a new store

When you add a new store under `src/`:

1. Import the helpers:
   ```ts
   import { inMemoryAllowed, isPostgresConfigured } from "../db/postgres";
   ```
2. Implement the Postgres path first.
3. Add an in-memory fallback only if dev/test coverage benefits from it; gate it
   behind `if (inMemoryAllowed()) { ... }`.
4. Add a final `throw new Error("DATABASE_URL is required for ...")` for the
   case where neither gate matches.
5. Add this store to the inventory table above.
6. Add unit tests for both paths.

## Verifying the audit

Quick checks an operator (or CI) can run:

- `git grep -E "inMemoryAllowed\(\)|isPostgresConfigured\(\)|isPostgresPersistenceEnabled\(\)" src/`
  should match every file in the inventory.
- `NODE_ENV=production node dist/index.js` without `DATABASE_URL` → exits with
  `PERSISTENCE_REQUIRED_ERROR`.
- `NODE_ENV=production DATABASE_URL=postgres://bogus:bogus@unreachable:5432/x
  node dist/index.js` → exits with `POSTGRES_UNREACHABLE_PRODUCTION_ERROR`
  after the connection check.
- `NODE_ENV=production AUTOFLOW_ALLOW_INMEMORY=true node dist/index.js` →
  exits with `PERSISTENCE_REQUIRED_ERROR` (the second gate doesn't override the
  first — production never falls back regardless).
- `NODE_ENV=test AUTOFLOW_ALLOW_INMEMORY=true npm test` → all tests pass using
  the in-memory paths.
